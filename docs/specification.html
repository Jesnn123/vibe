VIBE Format Specification

Version 1.0 - Complete Technical Reference

This document provides the complete technical specification for the VIBE configuration format, including formal grammar definitions, parsing algorithms, data type specifications, and implementation guidelines for library developers.

Overview

VIBE (Visual Intuitive Bracket Encoding) is a human-friendly configuration format designed to be more intuitive than YAML and more forgiving than JSON. It uses Python-like indentation for structure, supports both unquoted and quoted strings, and provides clear visual hierarchy through its bracket-based syntax.

Design Philosophy

VIBE prioritizes readability and ease of use while maintaining strict parsing rules that prevent ambiguity. Every valid VIBE document has exactly one interpretation.

Design Goals

Visual Hierarchy
Clear visual structure using brackets and indentation that mirrors the logical structure of data
Minimal Syntax
Reduced punctuation overhead with optional commas and minimal quoting requirements
Unambiguous Grammar
No edge cases or surprising behaviors - predictable parsing with clear error messages
Fast Parsing
Simple grammar enables efficient single-pass parsing with O(n) time complexity
Type Safety
Clear type inference rules with no implicit type coercion or surprising conversions
Human-Friendly
Intuitive syntax that feels natural to read and write, with helpful error messages

Grammar

Formal Grammar Definition (EBNF)

The VIBE format is defined by the following Extended Backus-Naur Form (EBNF) grammar:

EBNF 
document     = { assignment | comment | newline } ;
assignment  = identifier ":" value newline ;
value       = string | number | boolean | array | object ;
object      = "{" newline { assignment } "}" ;
array       = "[" [ value { "," value } ] "]" ;
identifier  = letter { letter | digit | "_" | "-" } ;

Tokens

Token Pattern Description Example
IDENTIFIER [a-zA-Z][a-zA-Z0-9_-]* Object keys and unquoted strings server_name
INTEGER -?[0-9]+ Whole numbers 42, -100
FLOAT -?[0-9]+\.[0-9]+ Decimal numbers 3.14, -2.5
BOOLEAN true|false Boolean values true, false
STRING "[^"]*" or '[^']*' Quoted strings "Hello World"
LBRACE { Object start {
RBRACE } Object end }
LBRACKET [ Array start [
RBRACKET ] Array end ]
COLON : Key-value separator :
COMMA , Array element separator ,
COMMENT #.* Single-line comment # Comment

Syntax Rules

Assignment

The fundamental unit of VIBE is the key-value assignment:

VIBE 
key value
name "John Doe"
age 30
active true
Key Naming Convention

Keys must start with a letter and can contain letters, numbers, underscores, and hyphens. Use descriptive names that clearly indicate the purpose of each value.

Object Declaration

Objects represent nested structured data using curly braces:

VIBE 
database {
    host localhost
    port 5432
    credentials {
        username admin
        password "secret123"
    }
}

Array Declaration

Arrays contain ordered collections of values using square brackets:

VIBE 
servers [web1 web2 web3]
ports [80 443 8080]

# Multi-line arrays for readability
allowed_origins [
    "https://example.com"
    "https://api.example.com"
    "https://admin.example.com"
]

Data Types

Type Inference

VIBE automatically infers types based on syntax. Numbers without decimal points are integers, numbers with decimals are floats, true and false are booleans, and everything else is treated as a string.

Integer Type

Integers represent whole numbers and are parsed as 64-bit signed values (int64).

VIBE 
positive 42
negative -100
zero 0
large 9223372036854775807
Property Value Notes
Range -2^63 to 2^63-1 int64 range
Pattern -?[0-9]+ Optional negative sign
Leading zeros Not allowed Except for 0 itself

Float Type

Floating-point numbers represent decimal values and are parsed as 64-bit IEEE 754 doubles.

VIBE 
pi 3.14159
negative -2.5
small 0.001
Decimal Point Required

A decimal point must be present for a number to be recognized as a float. Scientific notation (e.g., 1e10) is not supported in VIBE 1.0.

Boolean Type

Boolean values represent true/false states using the keywords true and false.

VIBE 
enabled true
debug false
production true

String Type

Unquoted Strings

Simple strings without special characters can be written without quotes:

VIBE 
name John
status active
role administrator
When to Use Unquoted Strings

Unquoted strings work great for simple identifiers, file names, and values without spaces or special characters. They make configs cleaner and more readable.

Quoted Strings

Strings with spaces, special characters, or that could be confused with other types must be quoted:

VIBE 
message "Hello, World!"
path "/usr/local/bin"
email "user@example.com"
number_as_string "42"

Supported escape sequences in quoted strings:

  • \\ - Backslash
  • \" - Double quote
  • \n - Newline
  • \t - Tab
  • \r - Carriage return

Arrays

Arrays are ordered collections of values enclosed in square brackets:

VIBE 
# Inline arrays
numbers [1 2 3 4 5]
names [Alice Bob Charlie]

# Multi-line arrays
colors [
    red
    green
    blue
]

# Mixed types (allowed but not recommended)
mixed [42 "text" true 3.14]

# Nested arrays
matrix [
    [1 2 3]
    [4 5 6]
    [7 8 9]
]
Array Syntax Rules
  • Elements are separated by commas
  • Trailing commas are allowed
  • Arrays can span multiple lines for readability
  • Empty arrays [] are valid
  • Nested arrays create multi-dimensional structures

Objects

Objects represent structured data with named fields enclosed in curly braces:

VIBE 
server {
    host localhost
    port 8080
    ssl {
        enabled true
        cert "/path/to/cert.pem"
        key "/path/to/key.pem"
    }
    routes [
        { path /api handler api_handler }
        { path /static handler static_handler }
    ]
}
Object Nesting Best Practices

Use indentation to clearly show hierarchy. Keep nesting levels reasonable (typically 3-4 levels max) for maintainability. Group related configuration together in logical objects.

Lexical Analysis

The lexical analyzer (lexer) processes the input character stream and produces a sequence of tokens. This is the first phase of parsing.

Character Classes

Class Pattern Usage
Letter [a-zA-Z] Identifier start
Digit [0-9] Numbers
Whitespace [ \t\r\n] Separation
Special :{}[], Syntax elements

Tokenization Process

  1. Skip whitespace - Space, tab, and newline characters are generally ignored except as token separators
  2. Identify token type - Determine what kind of token based on the first character(s)
  3. Consume characters - Read characters that belong to this token
  4. Create token object - Store the token type, value, and position information
  5. Repeat - Continue until end of file
Pseudocode 
function tokenize(input):
    tokens = []
    position = 0

    while position < input.length:
        skip whitespace

        if current char is letter:
            token = read_identifier_or_keyword()
        else if current char is digit or minus:
            token = read_number()
        else if current char is quote:
            token = read_string()
        else if current char is special:
            token = read_punctuation()
        else if current char is '#':
            skip_comment()
            continue

        tokens.append(token)

    return tokens

Comments

Comments in VIBE use the hash symbol (#) and extend to the end of the line:

VIBE 
# This is a full-line comment
server_name production  # Inline comment

# Multi-line comments require # on each line
# Like this
# And this

database {
    # Connection settings
    host localhost
    port 5432
}
Comment Guidelines
  • Document why, not what - Explain the reasoning behind configuration choices
  • Keep comments up to date with configuration changes
  • Use comments to mark sections in large configuration files
  • Avoid obvious comments that just restate the key name
  • Include URLs or references for complex configurations

Whitespace and Formatting

VIBE uses indentation for visual hierarchy but does not enforce significant whitespace like Python or YAML. The parser relies on brackets and braces for structure.

Whitespace is Visual Only

Unlike Python or YAML, indentation in VIBE does not affect parsing. Structure is determined entirely by brackets {} and braces []. This means you can format your config files however you prefer without worrying about breaking the parser.

Whitespace Rules

  • Spaces, tabs, and newlines separate tokens
  • Multiple whitespace characters are treated as one separator
  • Indentation is optional and purely cosmetic
  • Trailing whitespace is ignored
Recommended Style
  • Use 2-4 spaces for indentation
  • Be consistent within each file
  • Add blank lines between sections
  • One statement per line
  • Align nested structures visually
Valid But Not Recommended
  • No indentation at all
  • Mixing tabs and spaces
  • Inconsistent indentation levels
  • Multiple statements on one line
  • No blank lines between sections

Formatting Examples

VIBE - Recommended Style 
# Clean, readable formatting
server {
    host localhost
    port 8080

    database {
        connection "postgresql://localhost/mydb"
        pool_size 10
    }
}
VIBE - Valid But Messy 
# Still valid, but harder to read
server { host localhost port 8080
database { connection "postgresql://localhost/mydb"
  pool_size 10 } }
